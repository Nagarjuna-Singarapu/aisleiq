## Sidebar Features

### 1. Camera Filter

**Control:** Text input — _"Filter by Camera ID"_ at the top of the sidebar.

Scopes every chart, metric, and table on the main page to a single camera.

| Input | Effect |
|---|---|
| _(empty)_ | Shows aggregated data across all cameras |
| `CAM1` | Shows only data generated by the camera labelled `CAM1` |

The filter is applied on the next refresh cycle (immediate if Auto-refresh is on).

---

### 2. Auto-refresh

**Control:** Checkbox — _"Auto-refresh (10s)"_

When checked, the entire dashboard automatically reloads every 10 seconds, pulling the latest data from the API without any manual action.

- **Leave checked** while a processing job is running — you can watch occupancy, events, and job progress update live.
- **Uncheck** when you want to hold the current view for closer inspection without the page jumping.

---

### 3. Upload Video

**Control:** File uploader — _"📤 Upload Video"_ section in the sidebar.

Uploads a CCTV video file from your local machine directly into the server's `data/videos/` directory, making it immediately selectable for processing.

**Accepted formats:** `.mp4`, `.avi`, `.mov`, `.mkv`

**Behaviour:**
- Drag-and-drop or click **Browse files** to pick a file
- On upload, the file is written to `data/videos/` on the server
- A green ✅ confirmation message appears with the saved filename
- The file instantly appears in the **Process Video** dropdown below
- If a file with the same name already exists, a warning is shown and the existing file is kept (no overwrite)

---

### 4. Process Video

**Control:** Dropdown + inputs + button — _"▶ Process Video"_ section in the sidebar.

Submits a video file to the AI pipeline (detection → tracking → event generation → anomaly detection). Processing runs in a background thread so the dashboard stays responsive.

**Controls:**

| Control | What it does |
|---|---|
| **Select video file** | Dropdown of every video file currently in `data/videos/` |
| **Camera ID** | A free-text label assigned to this camera's data (e.g. `CAM1`, `entrance-cam`). Auto-filled from the filename. |
| **Frame skip (1=every frame, 10=fast)** | How many frames to skip between detections. `1` processes every frame (most accurate, slowest). `10` processes 1 in 10 frames (fastest, least accurate). Default is `3`. |
| **▶ Start Processing** | Submits the job. Shows a green toast with the Job ID on success, or a red error if the submission fails. Page reruns immediately so the new job appears in the Processing Jobs monitor. |

---

### 5. Refresh Now

**Control:** Button — _"🔄 Refresh Now"_ at the bottom of the sidebar.

Triggers an immediate page rerun, pulling the latest data from the API right now without waiting for the 10-second auto-refresh cycle.

## Main Page Features

### 6. API Status Banner

Displayed below the page title on every load.

| State | Appearance | Meaning |
|---|---|---|
| Online | ✅ Green banner — _"API online \| DB: ok \| v1.0.0"_ | API and database are reachable and healthy |
| Offline | ❌ Red banner + info message | API is not reachable; the rest of the page stops rendering until it comes back |

When the API is offline, the dashboard waits one refresh cycle and retries automatically — no manual reload needed.

---

### 7. KPI Metrics Row

Four metric cards displayed directly below the status banner.

| Metric | What it measures |
|---|---|
| **👥 Current Occupancy** | Estimated number of people inside the store right now (based on the most recently processed frame) |
| **🚶 Total Footfall** | Total unique individuals who have entered the store since processing began |
| **⏱ Avg Dwell (s)** | Average number of seconds a person spends inside the store |
| **🚨 Total Alerts** | Total anomaly alerts generated (crowd + loitering + queue combined) |

A caption below the cards shows the **peak occupancy count** and the exact time it was recorded.

All four metrics respect the **Camera Filter** — set a camera ID to see per-camera numbers.

---

### 8. Occupancy Timeline Chart

**Type:** Interactive Plotly line chart.  
**Data source:** `GET /analytics/occupancy`

Shows store occupancy (headcount) over time, sampled per minute.

| Axis | Meaning |
|---|---|
| X | Time |
| Y | Number of people in the store |

**Reading tips:**
- Sharp spikes = brief rush periods
- Broad peaks = sustained busy periods
- Flat sections near zero = quiet periods or gaps between video segments

Hover over any point to see the exact timestamp and occupancy count. Zoom and pan with the Plotly toolbar.

---

### 9. Zone Occupancy Bar Chart

**Type:** Interactive Plotly bar chart.  
**Data source:** `GET /analytics/zones`

Compares traffic distribution across the three store zones side by side.

| Zone | Default region of the video frame |
|---|---|
| `entrance` | Left 25% of the frame |
| `main_floor` | Centre 50% of the frame |
| `checkout` | Right 25% of the frame |

Each bar shows the total number of times anyone entered that zone. Use this to understand which parts of the store attract the most traffic — e.g. a high `checkout` count with many queue alerts means staffing needs attention there.

---

### 10. Dwell Time Histogram

**Type:** Interactive Plotly histogram.  
**Data source:** `GET /analytics/dwell-time`

Shows how long individual customers spend in the store, distributed into time buckets.

**Reading tips:**
- A spike in short buckets (0–30s) = many quick visits
- A long right tail = some customers staying a very long time (potential loitering)
- A narrow, tall distribution = consistent shopping pattern across customers

Hover over a bar to see the exact bucket range and count.

---

### 11. Recent Events Table

**Data source:** `GET /events?limit=50`  
**Location:** Below the charts, under _"📋 Recent Events"_

Displays the last 50 raw events generated during video processing, newest first.

| Column | Description |
|---|---|
| `event_id` | Unique identifier for this event |
| `camera_id` | Which camera generated it |
| `event_type` | The type of event (see table below) |
| `track_id` | The person's persistent track ID across frames |
| `zone` | Which store zone the person was in |
| `timestamp` | When the event occurred in the video timeline |
| `payload` | Extra context data (dwell seconds, occupancy count, zone names, etc.) |

**Event types shown in this table:**

| Event Type | When it fires |
|---|---|
| `person_entered` | A new confirmed person track appears |
| `person_exited` | A person's track ends (they left the frame/store) |
| `dwell_time_updated` | A person's dwell time crosses a reporting threshold |
| `zone_occupancy_changed` | A person moves from one zone to another |
| `crowd_detected` | Zone headcount exceeds the crowd threshold |
| `suspicious_loitering` | A person has been in one zone beyond the loitering threshold |
| `queue_detected` | Checkout zone headcount exceeds the queue threshold |

The table respects the **Camera Filter**.

---

### 12. Active Alerts Panel

**Data source:** `GET /alerts?acknowledged=false`  
**Location:** Below the events table, under _"🚨 Active Alerts"_

Shows all unacknowledged anomaly alerts as individual rows, colour-coded by severity.

| Colour | Severity | Alert type |
|---|---|---|
| 🔴 Red | `high` | Crowd detected |
| 🟠 Orange | `medium` | Loitering or Queue detected |

Each row shows:
- **Alert type** in bold (CROWD / LOITERING / QUEUE)
- **Message** — human-readable description (e.g. _"Crowd of 12 detected in main_floor"_)
- **Timestamp** — when the anomaly was detected
- **Acknowledge** button — marks the alert as reviewed and removes it from this panel

Once acknowledged, an alert no longer appears here but remains queryable via the API (`?acknowledged=true`).

If there are no active alerts, a green _"No active alerts"_ message is shown.

---

### 13. Processing Jobs Monitor

**Data source:** `GET /process-video/jobs`  
**Location:** Bottom of the main page — _"🔧 Processing Jobs"_ (expanded by default)

A live table showing every job that has ever been submitted, with real-time status.

| Column | Description |
|---|---|
| `job_id` | Unique numeric ID assigned when the job was submitted |
| `camera_id` | The camera label provided at submission |
| `status` | `pending` → `running` → `completed` / `failed` |
| `processed_frames` | Number of frames analysed so far |
| `total_frames` | Total frames in the source video |
| `fps` | Current processing throughput in frames per second |
| `created_at` | When the job was submitted |

**What to watch for:**
- `processed_frames` climbs in real time while `status = running`
- When `processed_frames = total_frames` and `status = completed`, all events and analytics are ready
- If `status = failed`, check the API terminal logs for the error
- Multiple jobs from different cameras can be tracked simultaneously in the same table
